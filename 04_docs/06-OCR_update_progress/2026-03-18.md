## time: 21:00

### Mục tiêu
GPU OOM prevention + E2E GPU stability & system resilience tests — đảm bảo worker không bị OOM khi khởi tạo model và xử lý liên tục.

### Files thay đổi

**New files:**
- `03_worker/app/utils/gpu_memory.py` — GPU memory management utilities:
  - `set_gpu_memory_fraction()`: Giới hạn GPU memory fraction trước khi load model (default 85%, configurable via `GPU_MEMORY_FRACTION` env)
  - `check_gpu_available(min_free_mb)`: Kiểm tra GPU memory trước khi init model
  - `log_gpu_memory(context)`: Log GPU allocated/reserved/free/total
  - `cleanup_gpu_memory()`: Aggressive cleanup — Python GC + `paddle.device.cuda.empty_cache()`
  - `get_gpu_memory_info()`: Trả về dict chi tiết GPU memory status
  - PaddlePaddle flags: `FLAGS_fraction_of_gpu_memory_to_use`, `FLAGS_initial_gpu_memory_in_mb=256`, `FLAGS_eager_delete_tensor_gb=0.0`

- `00_test/e2e/test_e2e_gpu_stability.py` — 15 GPU stability tests (GS-001→GS-015):
  - **Startup** (4 tests): health check, worker available, paddle_text responds, paddle_vl responds
  - **Sequential Processing** (4 tests): 5 images (detect slowdown), 3 PDFs, alternating image/PDF, 3 structured extractions
  - **Heavy Load** (3 tests): multi-page PDF stress, batch 3 files, structured PDF (heaviest GPU load)
  - **Cross-Engine** (3 tests): text→VL, VL→text, 5 rounds interleaved
  - **Post-Load** (1 test): health + smoke test after all previous tests

- `00_test/e2e/test_e2e_system_resilience.py` — 16 resilience tests (SR-001→SR-016):
  - **Error Recovery** (4 tests): invalid MIME, corrupt image, recovery after error, empty PDF
  - **Concurrent Uploads** (4 tests): rapid sequential, mixed formats, multi-user, upload during processing
  - **Unicode** (3 tests): Vietnamese full cycle, Vietnamese PDF, Vietnamese structured
  - **Service Stability** (3 tests): health consistency, service list consistency, request list after load
  - **Output Formats** (2 tests): all text formats, all structured formats

**Modified files:**
- `03_worker/app/engines/paddle_text/handler.py`:
  - Import `gpu_memory` utils
  - `__init__`: `set_gpu_memory_fraction()` + `check_gpu_available(400MB)` + `log_gpu_memory()` before/after init
  - `process()`: Dùng `cleanup_gpu_memory()` thay vì raw `paddle.device.cuda.empty_cache()`, thêm log sau processing

- `03_worker/app/engines/paddle_vl/handler.py`:
  - Import `gpu_memory` utils
  - `__init__`: `set_gpu_memory_fraction()` + `check_gpu_available(800MB)` (PPStructure cần nhiều hơn) + `log_gpu_memory()` before/after init
  - `process()`: Dùng `cleanup_gpu_memory()`, thêm log, final cleanup sau khi xử lý xong

- `03_worker/app/core/worker.py`:
  - Import `cleanup_gpu_memory`, `log_gpu_memory`
  - `process_job()` finally block: cleanup GPU memory + log sau mỗi job

### Lý do
**Vấn đề:** Worker GPU (đặc biệt PPStructureV3) load nhiều model cùng lúc (layout + table + OCR) có thể gây OOM. Xử lý PDF multi-page tích luỹ GPU memory. Không có monitoring hay giới hạn memory fraction.

**Giải pháp 3 lớp:**
1. **Prevention**: `FLAGS_fraction_of_gpu_memory_to_use=0.85` + `FLAGS_initial_gpu_memory_in_mb=256` giới hạn PaddlePaddle không chiếm hết GPU. Kiểm tra memory trước khi init model.
2. **Cleanup**: `cleanup_gpu_memory()` gọi GC + `empty_cache()` sau mỗi page và mỗi job. PPStructure VL handler có thêm final cleanup.
3. **Monitoring**: Log GPU memory tại 4 điểm: trước/sau init model, sau processing, sau mỗi job. Dễ dàng phát hiện memory leak qua logs.

**E2E tests** verify toàn bộ: worker startup OK (model loaded), sequential processing không leak, heavy PDF processing, cross-engine stability, error recovery không gây GPU leak.

Tổng E2E tests: 118 → **149 tests** (+15 GPU stability + 16 resilience).
