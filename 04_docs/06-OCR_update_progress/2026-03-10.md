## Bugfix: Service instance re-registration không reset trạng thái DEAD

### Triệu chứng

```
FAIL: 400 - {"detail":"No approved service available for method 'ocr_text_raw' tier 0"}
```

Service `ocr-paddle-vl` đã được admin approve nhưng upload vẫn bị từ chối. Test script cũng báo 404 khi check available services.

---

### Root Cause Analysis

**Bug 1 (Critical) — `ServiceInstanceRepository.register()` không reset status khi re-register**

Luồng gây ra bug:

```
Worker khởi động lần đầu → register() → instance WAITING
Admin approve → instance WAITING → ACTIVE           (OK)
Worker bị heartbeat monitor đánh dấu DEAD (timeout)
Worker restart → register() lại với cùng instance_id
  → existing record tìm thấy
  → chỉ update last_heartbeat_at
  → status vẫn giữ nguyên DEAD                     (BUG)
_validate_service_available() → 0 active instances → raise ServiceNotAvailable
```

File lỗi: `infrastructure/database/repositories/service_instance.py:62-68`

Code cũ:
```python
if existing:
    existing.last_heartbeat_at = datetime.now(timezone.utc)
    if metadata:
        existing.instance_metadata = json.dumps(metadata)
    return self.update(existing)  # ← status không được reset
```

**Bug 2 (Minor) — Test script gọi sai endpoint**

`05_test_cases/test_api_flow.py:38` gọi `GET /services` nhưng endpoint thực tế là `GET /services/available`.

---

### Fix

**Fix 1** — `repositories/service_instance.py`

Khi existing instance re-register và service type là APPROVED: reset status DEAD/WAITING → ACTIVE.

```python
if existing:
    existing.last_heartbeat_at = datetime.now(timezone.utc)
    if metadata:
        existing.instance_metadata = json.dumps(metadata)
    # Re-activation: if type is APPROVED, a re-registering instance should become ACTIVE
    if service_type.status == ServiceTypeStatus.APPROVED:
        if existing.status in (ServiceInstanceStatus.DEAD, ServiceInstanceStatus.WAITING):
            existing.status = ServiceInstanceStatus.ACTIVE
    return self.update(existing)
```

**Fix 2** — `05_test_cases/test_api_flow.py`

```python
# Trước
resp = client.get("/services")
# Sau
resp = client.get("/services/available")
```

---

### Files thay đổi

| File | Loại | Thay đổi |
|------|:----:|----------|
| `infrastructure/database/repositories/service_instance.py` | Fix | Re-registration reset status DEAD/WAITING → ACTIVE khi type APPROVED |
| `05_test_cases/test_api_flow.py` | Fix | Sửa endpoint `/services` → `/services/available` |

### Lưu ý vận hành

Nếu sau fix vẫn gặp lỗi tương tự, kiểm tra:

1. Worker container có đang chạy không:
   ```bash
   docker compose logs ocr-paddle-vl --tail=50
   ```
2. Xem danh sách instances qua admin:
   ```
   GET /api/v1/admin/service-instances
   ```
   Nếu list trống → worker chưa bao giờ gọi `/internal/register` → cần start worker.

3. Xác nhận service type có đúng `allowed_methods` và `allowed_tiers`:
   ```
   GET /api/v1/admin/service-types/{id}
   ```
   Phải có `"ocr_text_raw"` trong `allowed_methods` và `0` trong `allowed_tiers`.

## Thêm logging vào tất cả try/except blocks — Backend

### Mục tiêu

Đảm bảo mỗi except block đều emit log để dễ debug khi có lỗi trong production. Trước đây nhiều block dùng `pass` hoặc raise HTTPException mà không log gì.

---

### Files thay đổi — 8 files, 15 log calls

| File | Blocks | Log level | Ghi chú |
|------|:------:|-----------|---------|

| `api/v1/endpoints/jobs.py` | 4 | `DEBUG`, `WARNING` | Thêm logger. WARNING khi file không tìm thấy trong storage, DEBUG khi parse fail |
| `api/v1/endpoints/auth.py` | 2 | `INFO`, `WARNING` | Thêm logger. INFO cho user exists (expected), WARNING cho invalid credentials |
| `api/v1/endpoints/files.py` | 2 | `WARNING` | Thêm logger. WARNING khi original/result file không tìm thấy trong MinIO |

### Quy tắc log level áp dụng

| Level | Khi nào dùng |
|-------|-------------|
| `ERROR` + `exc_info=True` | Exception không mong đợi, cần stack trace để debug |
| `WARNING` | Lỗi business logic / resource không tìm thấy / thao tác thất bại |
| `INFO` | Sự kiện bình thường nhưng đáng ghi nhận (vd: user đã tồn tại khi register) |
| `DEBUG` | Parse fail có fallback, migration skip — không ảnh hưởng flow |
